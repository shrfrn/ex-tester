### Phase 2: Refactor validateIndentation() Function

**File:** `tests/validators/indentation.validator.js` (after split from Phase 1)

**Current issues:**
- 123 lines - far exceeds 30-line guideline
- Handles indent style detection, validation, and bracket tracking in one function
- Multiple nested loops and conditionals
- Difficult to test and maintain

**Refactoring approach - break into 4 functions:**

1. `validateIndentation(codeString)` - Main entry point (15-20 lines)
   - Split code into lines
   - Call helper functions
   - Aggregate results and return

2. `_detectIndentStyle(lines)` - Detect indent style and size (25-30 lines)
   - Iterate through lines to find first indented line
   - Determine if tabs or spaces
   - Calculate indent size for spaces
   - Return `{ indentStyle, indentSize }` or `null` if no indentation found

3. `_validateIndentConsistency(lines, indentStyle, indentSize, expectedLevels)` - Check consistency (20-25 lines)
   - Iterate through lines
   - Check if each line uses consistent style (tabs vs spaces)
   - Verify indent depth matches expected level
   - Return array of violations with line numbers and messages

4. `_calculateExpectedIndentLevels(lines)` - Bracket tracking logic (25-30 lines)
   - Iterate through lines
   - Track opening/closing brackets
   - Return array mapping line numbers to expected indent levels
   - Uses simplified bracket counting approach (as noted in current code comment)

### Phase 3: Fix Memory Leaks in Timer Mocks

**File:** `src/services/mock-browser.service.js`

**Current issue (lines 70-88):**
- `resetMocks()` function deletes interval/timeout objects from tracking dictionaries
- Sets `isActive: false` flag on objects before deletion
- Does NOT clear the actual Node.js timer IDs stored in `nodeIntervalId` and `nodeTimeoutId`
- Memory leak: Node.js timers continue to exist even after reset

**Fix approach:**

1. In `resetMocks()` function:
   - Before deleting `intervals[id]`, call `clearInterval(intervals[id].nodeIntervalId)` 
   - Before deleting `timeouts[id]`, call `clearTimeout(timeouts[id].nodeTimeoutId)`
   - This ensures Node.js timers are properly cleaned up

2. Add proper cleanup sequence:
   ```
   For each interval:
     - Set isActive = false
     - Call clearInterval(nodeIntervalId) 
     - Delete from intervals object
   
   For each timeout:
     - Set isActive = false
     - Call clearTimeout(nodeTimeoutId)
     - Delete from timeouts object
   ```

**Note:** The code already attempts this at lines 72-84, but the implementation has the logic inside the loop. Review shows it IS calling `clearInterval()` and `clearTimeout()`, so the actual issue may be more subtle - need to verify the timer IDs are valid when called.

### Phase 4: Improve Line Spacing Throughout Codebase

**Guideline violations found:**
- "Apply generous line spacing. Add a blank line after variable definitions."
- Need to add blank lines between logical blocks

**Files to modify:**

1. `src/services/test-runner.service.js` (lines 5-17, 20-40)
   - Add blank line after variable declarations
   - Add blank lines between logical blocks in try-catch-finally

2. `src/test-runner.js` (lines 47-88)
   - Add spacing between calculation blocks in `calculateStudentScores()`
   - Separate variable definitions from computation logic

3. Other service files to review and fix:
   - `src/services/sandbox.service.js`
   - `src/services/test-collector.service.js`
   - `src/services/type-checker.service.js`
   - `src/services/file-utils.service.js`
   - `src/services/util.service.js`

**Spacing rules to apply:**

1. Blank line after variable definitions (including const/let/var blocks)
2. Blank line before and after if/else blocks (except one-liners)
3. Blank line before and after loops (for/while)
4. Blank line before return statements (except one-line functions)
5. Chunk multiple variable definitions into logical groups with blank lines between groups
6. Blank line between function definitions

**Process:**
- Read each file
- Identify logical blocks
- Add spacing according to rules
- Verify no more than 4 consecutive non-empty lines (existing `validateLineSpacing` rule)
