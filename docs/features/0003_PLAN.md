# VM Sandbox Security Hardening

## Overview
Enhance the VM sandbox security to prevent student code from accessing dangerous Node.js APIs, executing arbitrary code, or behaving like malware. This addresses potential security vulnerabilities in the current sandbox implementation.

## Current Security Gaps

The sandbox currently blocks:
- `require`, `import` - Module loading
- `Promise`, `fetch`, `XMLHttpRequest` - Async/network operations

Missing protections:
- `eval()` and `Function` constructor - Direct code execution
- `process`, `global`, `Buffer` - Node.js internals
- `__dirname`, `__filename`, `module`, `exports` - File system exposure
- Prototype pollution vectors
- Additional network/storage APIs

## Implementation Details

### Blocked API Categories

**1. Code Execution Prevention**
- `eval()` - Direct string-to-code execution
- `Function()` - Constructor that creates functions from strings
- `GeneratorFunction`, `AsyncFunction`, `AsyncGeneratorFunction` - Alternative function constructors

**2. Node.js Internals Protection**
- `process` - Access to exit(), env vars, system info
- `global` - Node.js global object (like window in browser)
- `Buffer` - Binary data operations, potential memory issues

**3. Module System Isolation**
- `require`, `import` - Already blocked, keep
- `__dirname`, `__filename` - File path exposure
- `module`, `exports` - Module system manipulation

**4. Network/Async Prevention**
- `Promise`, `fetch`, `XMLHttpRequest` - Already blocked, keep
- `WebSocket` - Websocket connections
- `EventSource` - Server-sent events
- `Worker`, `SharedWorker` - Background threads

**5. Storage Prevention**
- `localStorage`, `sessionStorage` - Browser storage
- `indexedDB`, `openDatabase` - Database access

**6. Prototype Pollution Prevention**
- Block `Object.setPrototypeOf()`
- Block `__proto__` access
- Freeze built-in prototypes at initialization

```javascript
    })
}
```

## Migration Notes

**Breaking Changes:**
- None - only adds restrictions

**Compatibility:**
- Student code using legitimate JavaScript will be unaffected
- Only blocks dangerous/malicious patterns
- Existing tests should pass unchanged

## Testing Strategy

1. **Positive Tests** - Verify legitimate student code still works:
   - Functions, loops, arrays, objects
   - Math operations
   - String manipulation
   - Mocked browser APIs (prompt, alert, console)

2. **Negative Tests** - Verify dangerous code is blocked:
   - Attempt to use eval()
   - Attempt to access process
   - Attempt to require modules
   - Attempt prototype pollution
   - Attempt to access file system

3. **Timeout Tests** - Verify timeout protection:
   - Infinite while loop
   - Infinite recursion
   - Missing prompt responses in sentinel loop

## Success Criteria

- All dangerous APIs return `undefined` or throw errors when accessed
- Built-in prototypes are frozen and immutable
- Timeout protection prevents hanging execution
- Legitimate student code executes normally
- Security tests pass for all blocked APIs
- No performance degradation from additional security measures
